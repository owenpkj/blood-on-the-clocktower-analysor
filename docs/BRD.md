# 血染钟楼分析工具 - 业务需求文档 (BRD)

## 问题陈述

血染钟楼是一款复杂的社交推理桌游，善良阵营玩家需要通过收集信息、逻辑推理来识别隐藏的邪恶玩家。然而：

1. **信息量大**：随着游戏进行，玩家需要记住大量的角色声称、投票行为、死亡信息等
2. **推理复杂**：需要同时考虑多种可能的"世界状态"（谁是邪恶、谁在醉酒/中毒）
3. **时间压力**：白天讨论时间有限，难以进行充分的逻辑推演
4. **认知负担**：人脑难以同时维护多个平行假设并追踪其一致性

善良阵营玩家需要一个工具来辅助记录信息、进行逻辑推理，从而提高找出邪恶玩家的成功率。

## 目标

### 主要目标
1. **信息管理**：帮助玩家系统化记录游戏中的所有关键信息
2. **逻辑推理**：基于已知信息，推导出最可能的"世界模型"（角色分配）
3. **决策支持**：为玩家的投票和处决决策提供参考依据

### 成功指标
- 用户能够在30秒内完成单个白天的信息录入
- 系统能够在每次信息更新后生成合理的世界模型推测
- 推理结果具有可解释性，用户能够理解推理依据

## 目标用户

**主要用户**：血染钟楼善良阵营玩家

**用户特征**：
- 了解血染钟楼基本规则
- 希望提高游戏胜率
- 愿意在游戏过程中使用辅助工具
- 不会欺骗工具（输入的信息是主观真实的）

**用户场景**：
- 线下游戏时使用手机/平板辅助
- 线上游戏时使用电脑辅助
- 游戏复盘分析

## 范围

### 在范围内 (In Scope)

**准备阶段**
- 选择游戏剧本
- 输入游戏人数
- 输入玩家自己的座位号
- 输入玩家自己的角色

**游戏阶段 - 信息记录**
- 记录每夜死亡情况
- 记录每个玩家的角色声称
- 记录每个玩家声称获得的信息
- 记录提名情况（谁提名谁）
- 记录投票情况（哪些号码投票）
- 记录处决结果
- 记录玩家自己获得的信息（作为可信基准，但可能受醉酒/中毒影响）

**推理阶段**
- 从第3个白天开始提供推理结果
- 展示概率最高的3个世界模型
- 每个世界模型包含：置信度、角色分配、关键推理节点
- 每次信息更新后刷新推理结果

**验证机制**
- 主推理Agent生成候选世界模型
- 验证Agent对世界模型进行逻辑核验和规则检查
- 验证通过后展示给用户

**剧本支持**
- 初出茅庐（社区自制）
- 暗流涌动（官方第一剧本）
- 意气用事（社区自制）
- 杀手往事（社区自制）

### 不在范围内 (Out of Scope)

**MVP阶段不包含**
- 用户对世界模型的二次追问/深度交互
- 邪恶阵营视角支持
- 多人协作/信息共享
- 游戏历史记录和统计
- 自定义剧本支持
- 说书人辅助功能
- 移动端原生App

**永久不包含**
- 自动化游戏操作（如自动投票）
- 与线上游戏平台的集成
- 实时语音/视频分析

## 功能需求

### FR1: 游戏初始化
- FR1.1: 用户可以从预设列表中选择剧本
- FR1.2: 用户可以输入游戏总人数（5-15人）
- FR1.3: 系统根据人数自动计算角色配置（镇民/外来者/爪牙/恶魔数量）
- FR1.4: 用户可以输入自己的座位号（1-N）
- FR1.5: 用户可以从剧本角色列表中选择自己的角色
- FR1.6: 系统初始化游戏状态，进入第1天

### FR2: 每日信息记录
- FR2.1: 用户可以记录昨夜死亡的玩家座位号（支持多人死亡和平安夜）
- FR2.2: 用户可以为每个玩家添加角色声称
- FR2.3: 用户可以为每个玩家添加其声称的信息内容
- FR2.4: 用户可以记录提名事件（提名者 → 被提名者）
- FR2.5: 用户可以记录每次提名的投票者列表
- FR2.6: 用户可以记录当日处决结果（被处决者座位号或无人被处决）
- FR2.7: 用户可以记录自己本轮获得的信息
- FR2.8: 用户可以推进到下一天
- FR2.9: 系统维护完整的游戏时间线

### FR3: 推理引擎
- FR3.1: 从第3个白天开始，系统自动触发推理
- FR3.2: 主推理Agent基于所有已知信息生成候选世界模型
- FR3.3: 验证Agent对候选世界模型进行逻辑核验
- FR3.4: 验证Agent检查世界模型是否符合游戏规则
- FR3.5: 主推理Agent根据验证反馈修正世界模型
- FR3.6: 系统展示Top 3世界模型

### FR4: 世界模型展示
- FR4.1: 每个世界模型显示置信度（百分比或等级）
- FR4.2: 每个世界模型显示所有玩家的推测角色
- FR4.3: 每个世界模型显示关键推理节点（为什么得出这个结论）
- FR4.4: 世界模型按置信度降序排列
- FR4.5: 每次用户输入新信息后，世界模型自动刷新

### FR5: 信息状态管理
- FR5.1: 系统区分"用户自己的信息"和"其他玩家声称的信息"
- FR5.2: 用户自己的信息标记为高可信度（但考虑醉酒/中毒可能）
- FR5.3: 其他玩家的信息标记为待验证
- FR5.4: 系统追踪每条信息的来源和时间点

## 非功能需求

### NFR1: 性能
- NFR1.1: 信息录入操作响应时间 < 500ms
- NFR1.2: 推理结果生成时间 < 30秒
- NFR1.3: 页面首次加载时间 < 3秒

### NFR2: 可用性
- NFR2.1: 界面简洁，支持快速信息录入
- NFR2.2: 支持移动端浏览器访问
- NFR2.3: 关键操作有确认提示，防止误操作
- NFR2.4: 支持撤销最近一次操作

### NFR3: 可靠性
- NFR3.1: 游戏状态本地持久化，刷新页面不丢失
- NFR3.2: 推理失败时显示友好错误信息
- NFR3.3: 网络中断时保留已录入数据

### NFR4: 可维护性
- NFR4.1: 剧本数据与代码分离，便于添加新剧本
- NFR4.2: 推理Prompt可配置，便于优化
- NFR4.3: 代码结构清晰，有基本注释

## 约束与假设

### 约束
- C1: 用户需要自行提供LLM API访问能力（或使用预配置的后端服务）
- C2: 推理质量依赖于LLM的能力
- C3: 工具不能保证推理结果100%正确

### 假设
- A1: 用户是善良阵营玩家
- A2: 用户不会故意输入虚假信息
- A3: 用户了解血染钟楼的基本规则
- A4: 用户在游戏过程中有条件使用电子设备
- A5: 用户输入的"自己获得的信息"是主观真实的（即用户相信这是真的，但可能因醉酒/中毒而实际错误）

## 术语表

| 术语 | 定义 |
|------|------|
| 世界模型 | 一种可能的游戏状态假设，包括每个玩家的真实角色、谁在醉酒/中毒等 |
| 置信度 | 某个世界模型为真的概率估计 |
| 角色声称 | 玩家公开声明自己是什么角色 |
| 信息声称 | 玩家公开声明自己获得了什么信息 |
| 可信基准 | 用户自己获得的信息，假设用户不欺骗工具，但可能受游戏机制影响而错误 |
| 推理节点 | 推理过程中的关键逻辑步骤 |
| 主推理Agent | 负责生成世界模型的LLM Agent |
| 验证Agent | 负责检验世界模型逻辑和规则合规性的LLM Agent |
